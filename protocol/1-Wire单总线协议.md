# 1-Wire 单总线协议

## 概述

1-Wire 是由达拉斯半导体（Dallas Semiconductor，现 Maxim）开发的一种**单线、半双工**串行通讯协议。顾名思义，它只需一根数据线（加 GND）即可完成通讯，甚至支持**寄生供电**（从数据线上取电，连 VCC 线都省了）。

- 通讯方式：异步、半双工、单主多从
- 所需引脚：**1根**数据线（DQ）+ GND
- 寄生供电：可选，设备直接从 DQ 线取电
- 速度：标准模式 15.4 kbps，超速模式 125 kbps
- 典型应用：DS18B20 温度传感器、DS2431 EEPROM、iButton

---

## 硬件连接

### 外部供电模式（推荐）

```
MCU                        DS18B20
  DQ ──────┬────────────── DQ（数据）
           │
          4.7kΩ 上拉电阻
           │
          VCC（3.3V 或 5V）──── VDD（供电）
  GND ─────────────────── GND
```

### 寄生供电模式（省一根线）

```
MCU                        DS18B20
  DQ ──────┬────────────── DQ（数据 + 供电）
           │
          4.7kΩ 上拉电阻
           │
          VCC（3.3V 或 5V）
  GND ─────────────────── GND
                           VDD ── GND（VDD 接地，从DQ线取电）
```

> 寄生供电时，DS18B20 在 DQ 高电平期间通过内部电容储能，低电平期间消耗储存的能量。

### 多设备挂载（同一根线）

```
MCU
  DQ ──────┬──────────────┬──────────────┬──────
           │              │              │
          4.7kΩ         DS18B20_1     DS18B20_2     DS18B20_3
           │
          VCC
```

> 所有设备共享同一根 DQ 线，通过 64 位唯一 ROM 地址区分。

---

## 通讯基本结构

每一次完整的 1-Wire 通讯，都遵循固定的三步结构：

```
第一步：复位 + 存在检测
第二步：ROM 命令（选择目标设备）
第三步：功能命令（读写数据）
```

这三步缺一不可，顺序不能颠倒。下面先介绍底层时序，再结合实际例子串联整个流程。

---

## 底层时序信号详解

1-Wire 所有通讯都基于三种基本时序：复位脉冲、写时隙、读时隙。

### 1. 复位脉冲（Reset Pulse）+ 存在脉冲（Presence Pulse）

每次通讯开始前，主机必须先发复位脉冲，从机回应存在脉冲，告知主机"我在线"。

```
时间轴（μs）：
  0        480      540  560       640
  │         │        │    │         │
主机：
  ▔▔▔▔▔▔▔▔▔▔         ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
         ▔▔▔▔▔▔▔▔▔▔▔▔▔
  高      拉低480μs   释放（高）
  
从机：
  ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
                              ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
                         检测到复位后，拉低60~240μs（存在脉冲）
```

**时序要求：**
| 阶段 | 时间 |
|------|------|
| 主机拉低（复位脉冲） | 480μs 以上 |
| 主机释放后等待 | 15~60μs |
| 从机拉低（存在脉冲） | 60~240μs |
| 主机采样窗口 | 释放后 60μs 内检测低电平 |

**判断结果：**
- 检测到低电平 → 总线上有从机，可以继续通讯
- 未检测到低电平 → 无从机响应，通讯终止

---

### 2. 写时隙（Write Slot）

主机向从机发送 1 bit 数据，分为"写 1"和"写 0"两种时隙，每个时隙最短 60μs。

#### 写 1（Write 1 Slot）

```
时间轴（μs）：
  0    1~15          60
  │      │            │
  ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
       ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
  高   拉低1~15μs     释放（高），从机在15μs内采样读到"1"
```

#### 写 0（Write 0 Slot）

```
时间轴（μs）：
  0                  60
  │                   │
  ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
       ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
  高   拉低保持60μs    释放，从机在15μs内采样读到"0"
```

**关键区别：**
- 写 1：主机拉低后迅速释放（< 15μs），从机采样时总线已回高
- 写 0：主机拉低保持整个时隙（≥ 60μs），从机采样时总线仍为低

---

### 3. 读时隙（Read Slot）

主机读取从机发来的 1 bit 数据。主机先拉低触发，然后释放，由从机决定总线电平。

```
时间轴（μs）：
  0  1~15   15        60
  │    │     │         │
主机：
  ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
       ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
  高   拉低1μs  释放，然后在15μs处采样

从机发送"1"：
  ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
  不拉低，总线保持高 → 主机采样读到"1"

从机发送"0"：
  ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
       ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
  从机拉低保持15μs以上 → 主机采样读到"0"
```

> 数据传输顺序：**低位（LSB）先发**，即 bit0 最先传输。

## ROM 命令（第二步）

复位成功后，主机必须发送一条 ROM 命令，用于寻址或管理总线上的从机。

| 命令 | 代码 | 说明 |
|------|------|------|
| 搜索 ROM | `0xF0` | 枚举总线上所有设备的 64 位 ROM 地址 |
| 读取 ROM | `0x33` | 仅适用于总线上只有一个设备时，直接读取其 ROM 地址 |
| 匹配 ROM | `0x55` | 发送 64 位地址，精确选中某一个设备 |
| 跳过 ROM | `0xCC` | 跳过地址匹配，广播给所有设备（总线只有一个设备时常用） |
| 报警搜索 | `0xEC` | 只有处于报警状态的设备才响应 |

### 64 位 ROM 地址结构

每个 1-Wire 设备出厂时都烧录了全球唯一的 64 位地址：

```
 bit63      bit56  bit55              bit8  bit7       bit0
┌──────────────┬──────────────────────────┬──────────────┐
│  8位 CRC校验  │      48位 序列号（唯一）   │  8位 家族码  │
└──────────────┴──────────────────────────┴──────────────┘
```

- **家族码**：标识设备类型，DS18B20 的家族码为 `0x28`
- **序列号**：48 位，保证全球唯一
- **CRC**：对前 56 位的校验，用于验证数据完整性

## 完整通讯示例：读取 DS18B20 温度

DS18B20 是最常见的 1-Wire 温度传感器，下面以"读取一个 DS18B20 的温度"为例，完整演示整个通讯过程。

### 场景设定

- 总线上只有 **1 个** DS18B20
- 使用**跳过 ROM**命令（省去地址匹配步骤）
- 读取温度需要两次完整的通讯序列

---

### 第一次通讯序列：启动温度转换

**目的：** 命令 DS18B20 开始测量温度，结果存入内部暂存器。

```
步骤 1：主机发复位脉冲
─────────────────────────────────────────────────────
主机拉低 DQ 线，保持 480μs
主机释放 DQ 线
等待 15~60μs 后采样
DS18B20 拉低 DQ 线 60~240μs（存在脉冲）
主机检测到低电平 → 确认 DS18B20 在线 ✓

步骤 2：主机发送 ROM 命令 —— 跳过 ROM（0xCC）
─────────────────────────────────────────────────────
发送 8 bit，LSB 先发：
  0xCC = 1100 1100（二进制）
  发送顺序：0, 0, 1, 1, 0, 0, 1, 1

  bit0=0 → 写0时隙：主机拉低60μs
  bit1=0 → 写0时隙：主机拉低60μs
  bit2=1 → 写1时隙：主机拉低1μs后释放
  bit3=1 → 写1时隙：主机拉低1μs后释放
  bit4=0 → 写0时隙：主机拉低60μs
  bit5=0 → 写0时隙：主机拉低60μs
  bit6=1 → 写1时隙：主机拉低1μs后释放
  bit7=1 → 写1时隙：主机拉低1μs后释放

步骤 3：主机发送功能命令 —— 启动温度转换（0x44）
─────────────────────────────────────────────────────
发送 8 bit，LSB 先发：
  0x44 = 0100 0100（二进制）
  发送顺序：0, 0, 1, 0, 0, 0, 1, 0

  bit0=0 → 写0时隙
  bit1=0 → 写0时隙
  bit2=1 → 写1时隙
  bit3=0 → 写0时隙
  bit4=0 → 写0时隙
  bit5=0 → 写0时隙
  bit6=1 → 写1时隙
  bit7=0 → 写0时隙

DS18B20 开始内部温度转换（12位精度约需 750ms）
此时主机可以轮询 DQ 线：读到 0 表示转换中，读到 1 表示转换完成
```

---

### 第二次通讯序列：读取温度数据

**目的：** 读取 DS18B20 暂存器中的 9 字节数据（含温度值）。

```
步骤 1：主机再次发复位脉冲
─────────────────────────────────────────────────────
（同上，重新握手确认设备在线）

步骤 2：主机发送 ROM 命令 —— 跳过 ROM（0xCC）
─────────────────────────────────────────────────────
（同上，发送 0xCC）

步骤 3：主机发送功能命令 —— 读暂存器（0xBE）
─────────────────────────────────────────────────────
发送 8 bit，LSB 先发：
  0xBE = 1011 1110（二进制）
  发送顺序：0, 1, 1, 1, 1, 1, 0, 1

步骤 4：主机连续读取 9 字节（72 bit）
─────────────────────────────────────────────────────
主机发起 72 个读时隙，DS18B20 依次输出：

  字节0（温度低字节）：假设读到 0x50
  字节1（温度高字节）：假设读到 0x05
  字节2：TH 寄存器（报警上限）
  字节3：TL 寄存器（报警下限）
  字节4：配置寄存器（分辨率）
  字节5~7：保留字节
  字节8：CRC 校验值
```

---

### 温度数据解析

读到的原始温度数据为 16 位（字节1 高字节 + 字节0 低字节）：

```
字节1（高字节）：0x05 = 0000 0101
字节0（低字节）：0x50 = 0101 0000

拼合为 16 位：0000 0101 0101 0000 = 0x0550

DS18B20 温度寄存器格式（12位分辨率）：
 bit15~bit11：符号位（0=正温度，1=负温度）
 bit10~bit4 ：整数部分
 bit3~bit0  ：小数部分（精度 0.0625°C/bit）

0x0550 = 0000 0101 0101 0000
符号位 = 0（正温度）
数值   = 0x0550 = 1360（十进制）
温度   = 1360 × 0.0625 = 85.0°C  ← 这是 DS18B20 上电默认值

实际例子：读到 0x0191
0x0191 = 0000 0001 1001 0001
数值   = 0x0191 = 401（十进制）
温度   = 401 × 0.0625 = 25.0625°C ≈ 25.06°C
```

## 多设备通讯示例：精确寻址

当总线上挂载多个 DS18B20 时，必须使用**匹配 ROM（0x55）**命令精确选中目标设备。

### 场景设定

总线上有两个 DS18B20：
- 设备 A：ROM 地址 = `28 FF 64 1E 3A 04 00 A1`（十六进制，低字节在前）
- 设备 B：ROM 地址 = `28 FF 64 1E 3A 04 00 B2`

现在只想读取**设备 A** 的温度。

### 通讯流程

```
【第一次通讯：只让设备A启动温度转换】

步骤 1：复位 + 存在检测
  主机发复位脉冲
  设备A 和 设备B 都回应存在脉冲（两个设备都在线）

步骤 2：发送 匹配ROM 命令（0x55）
  主机发送：0x55

步骤 3：发送 64 位 ROM 地址（选中设备A）
  主机依次发送 8 字节：
  0x28, 0xFF, 0x64, 0x1E, 0x3A, 0x04, 0x00, 0xA1

  此时：
  ✓ 设备A：地址匹配，进入激活状态，准备接收功能命令
  ✗ 设备B：地址不匹配，进入等待状态，忽略后续命令

步骤 4：发送功能命令 —— 启动温度转换（0x44）
  只有设备A执行温度转换，设备B不响应

【等待 750ms】

【第二次通讯：读取设备A的温度】

步骤 1：复位 + 存在检测
步骤 2：发送 匹配ROM（0x55）+ 设备A的64位地址
步骤 3：发送 读暂存器命令（0xBE）
步骤 4：读取 9 字节温度数据
```

## 寄生供电下的特殊处理

温度转换是高耗电操作，寄生供电模式下 DQ 线需要提供强上拉电流。

```
发送 0x44 命令后，立即将 DQ 线强制拉高（不能释放）：

DQ: ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔
     │←── 0x44 ──→│←── 强上拉保持 750ms（转换期间不能有其他通讯）──→│

实现方式：
  - 使用 MOSFET 将 DQ 线强制连接到 VCC
  - 或者将 MCU 的 GPIO 切换为推挽输出模式，输出高电平
  - 转换完成后恢复开漏模式
```

## 错误处理与 CRC 校验

### CRC 校验

读取暂存器的第 9 字节是前 8 字节的 CRC-8 校验值（多项式 `x⁸ + x⁵ + x⁴ + 1`，即 `0x31`）。

```
读到的 9 字节：50 05 4B 46 7F FF 0C 10 1C

前8字节：50 05 4B 46 7F FF 0C 10
计算CRC：0x1C
第9字节：0x1C

CRC 匹配 ✓ → 数据有效
CRC 不匹配 ✗ → 数据损坏，需要重新读取
```

### 常见错误及处理

| 现象 | 可能原因 | 处理方式 |
|------|----------|----------|
| 复位后无存在脉冲 | 设备未连接、上拉电阻缺失、电源不足 | 检查硬件连接和上拉电阻 |
| 读到全 0xFF | 总线浮空，无设备拉低 | 检查 DQ 线连接 |
| 读到全 0x00 | 总线被持续拉低（短路） | 检查是否有设备损坏 |
| CRC 校验失败 | 信号干扰、时序不准确 | 重试读取，检查时序精度 |
| 温度值异常（85°C） | 转换未完成就读取 | 等待足够的转换时间（≥750ms） |

## 功能命令速查（DS18B20）

| 命令 | 代码 | 说明 |
|------|------|------|
| 启动温度转换 | `0x44` | 开始 A/D 转换，结果存入暂存器 |
| 读暂存器 | `0xBE` | 读取 9 字节暂存器数据 |
| 写暂存器 | `0x4E` | 写入 TH、TL 和配置寄存器 |
| 复制暂存器 | `0x48` | 将暂存器数据保存到 EEPROM |
| 召回 EEPROM | `0xB8` | 从 EEPROM 恢复数据到暂存器 |
| 读供电方式 | `0xB4` | 查询设备是否使用寄生供电 |

## 分辨率与转换时间

DS18B20 支持 9~12 位分辨率，通过配置寄存器（字节4）设置：

| 分辨率 | 精度 | 最大转换时间 | 配置寄存器 |
|--------|------|-------------|-----------|
| 9 位  | 0.5°C    | 93.75ms | `0x1F` |
| 10 位 | 0.25°C   | 187.5ms | `0x3F` |
| 11 位 | 0.125°C  | 375ms   | `0x5F` |
| 12 位 | 0.0625°C | 750ms   | `0x7F`（默认）|

## 代码示例（伪代码）

```c
// 读取单个 DS18B20 温度（外部供电，总线只有一个设备）

float DS18B20_ReadTemperature(void) {
    uint8_t data[9];
    int16_t raw;

    // ── 第一次通讯：启动温度转换 ──
    OneWire_Reset();           // 复位 + 等待存在脉冲
    OneWire_WriteByte(0xCC);   // 跳过 ROM
    OneWire_WriteByte(0x44);   // 启动温度转换

    delay_ms(750);             // 等待 12 位转换完成

    // ── 第二次通讯：读取温度 ──
    OneWire_Reset();           // 再次复位
    OneWire_WriteByte(0xCC);   // 跳过 ROM
    OneWire_WriteByte(0xBE);   // 读暂存器

    for (int i = 0; i < 9; i++) {
        data[i] = OneWire_ReadByte();  // 读取 9 字节
    }

    // CRC 校验
    if (CRC8(data, 8) != data[8]) {
        return -999.0f;  // 数据错误
    }

    // 解析温度
    raw = (int16_t)((data[1] << 8) | data[0]);
    return raw * 0.0625f;  // 12 位精度：每 bit = 0.0625°C
}
```

---

## 总结

1-Wire 通讯的核心逻辑可以归纳为：

```
每次通讯 = 复位握手 → ROM命令（寻址）→ 功能命令（操作）→ 数据读写
```

- 所有时序都靠**精确的时间控制**实现，MCU 需要关闭中断或使用硬件定时器保证精度
- 数据传输**低位（LSB）先行**
- 多设备场景必须通过 **64 位 ROM 地址**区分，不能省略寻址步骤
- 读完数据务必做 **CRC 校验**，确保数据可靠性
