# STM32CubeIDE 项目系统文件说明

本文档介绍 STM32CubeIDE 自动生成的系统框架文件的作用，适用于 STM32F103C8T6（Blue Pill）项目。

---

## 启动流程总览

芯片上电后，代码的执行顺序如下：

```
上电/复位
    │
    ▼
startup_stm32f103c8tx.s   ← 汇编启动文件，最先执行
    │  1. 设置栈指针 SP
    │  2. 调用 SystemInit()
    │  3. 初始化 .data / .bss 段
    │  4. 调用 main()
    ▼
system_stm32f1xx.c        ← SystemInit() 在这里定义
    │  配置向量表位置
    ▼
main.c                    ← 用户应用入口
    │  HAL_Init()  →  stm32f1xx_hal_msp.c（底层硬件初始化）
    │  SystemClock_Config()
    │  MX_XXX_Init()
    ▼
运行中产生中断
    ▼
stm32f1xx_it.c            ← 中断服务程序在这里处理
```

---

## 文件逐一说明

### 1. `Core/Startup/startup_stm32f103c8tx.s`

**汇编启动文件，芯片复位后第一个执行的代码。**

| 职责 | 说明 |
|------|------|
| 初始化栈指针 | 将 SP 设置为链接脚本定义的 `_estack`（RAM 末尾） |
| 调用 SystemInit | 在跳转 main 之前先初始化时钟 |
| 初始化 .data 段 | 将 Flash 中存储的初始值复制到 RAM |
| 清零 .bss 段 | 将未初始化的全局变量清零 |
| 定义中断向量表 | `g_pfnVectors` 包含所有异常和外设中断的入口地址 |
| 跳转到 main | 调用 `__libc_init_array` 后再调用 `main()` |

> **不需要修改。** 除非需要重定位向量表或修改启动行为，否则保持原样。

所有中断处理函数都以 `.weak` 方式声明，默认指向 `Default_Handler`（死循环）。一旦你在 `stm32f1xx_it.c` 中实现了同名函数，链接器会自动用你的实现覆盖默认行为。

---

### 2. `Core/Src/system_stm32f1xx.c`

**CMSIS 系统初始化文件，由 ST 官方提供。**

提供三个核心内容：

| 内容 | 说明 |
|------|------|
| `SystemInit()` | 启动文件调用，配置向量表位置（本项目保持默认） |
| `SystemCoreClockUpdate()` | 根据当前时钟寄存器值更新 `SystemCoreClock` 变量 |
| `SystemCoreClock` 变量 | 全局变量，记录当前 HCLK 频率，HAL 库内部用来计算延时等 |

> **通常不需要修改。** 时钟配置的实际工作由 `main.c` 中的 `SystemClock_Config()` 完成，`system_stm32f1xx.c` 只做最底层的初始化。

---

### 3. `Core/Src/stm32f1xx_hal_msp.c`

**HAL MSP（MCU Support Package）文件，外设硬件初始化的回调钩子。**

HAL 库的设计原则是将"外设逻辑"和"底层硬件"分离：

- HAL 库负责外设逻辑（协议、时序、状态机）
- `_msp.c` 负责底层硬件（GPIO 复用、时钟使能、DMA 绑定、NVIC 配置）

**调用关系示例：**

```c
// 你调用：
HAL_UART_Init(&huart1);

// HAL 库内部会自动回调：
void HAL_UART_MspInit(UART_HandleTypeDef *huart) {
    // 你在这里开 UART 时钟、配置 TX/RX 引脚
}
```

**当前项目中已有的内容：**

```c
void HAL_MspInit(void) {
    __HAL_RCC_AFIO_CLK_ENABLE();   // 开启复用功能时钟
    __HAL_RCC_PWR_CLK_ENABLE();    // 开启电源管理时钟
    __HAL_AFIO_REMAP_SWJ_NOJTAG(); // 禁用 JTAG，仅保留 SWD
                                    // 释放 PA15/PB3/PB4 给普通 GPIO 使用
}
```

> **会随外设增加而扩展。** 每次通过 CubeMX 添加 UART、SPI、I2C 等外设，对应的 `HAL_XXX_MspInit()` 和 `HAL_XXX_MspDeInit()` 会自动生成到此文件中。

---

### 4. `Core/Src/stm32f1xx_it.c` / `Core/Inc/stm32f1xx_it.h`

**中断服务程序（ISR）文件。**

所有中断处理函数都在这里实现。当前项目包含：

| 函数 | 类型 | 说明 |
|------|------|------|
| `NMI_Handler` | CPU 异常 | 不可屏蔽中断，卡死在 `while(1)` |
| `HardFault_Handler` | CPU 异常 | 硬件错误（访问非法地址、除零等），卡死在 `while(1)` |
| `MemManage_Handler` | CPU 异常 | 内存管理错误 |
| `BusFault_Handler` | CPU 异常 | 总线错误 |
| `UsageFault_Handler` | CPU 异常 | 指令使用错误（未定义指令等） |
| `SVC_Handler` | 系统调用 | RTOS 中使用，裸机下为空 |
| `DebugMon_Handler` | 调试监视器 | 调试相关，通常为空 |
| `PendSV_Handler` | 可悬挂系统调用 | RTOS 任务切换使用，裸机下为空 |
| `SysTick_Handler` | **系统心跳** | **每 1ms 调用一次 `HAL_IncTick()`，HAL 延时系统依赖此函数** |

> **SysTick_Handler 是最关键的一个**，`HAL_Delay()`、超时检测、`uwTick` 计数全部依赖它。不能删除或修改。

> **会随外设中断增加而扩展。** 使能 TIM、UART 中断后，`TIM2_IRQHandler`、`USART1_IRQHandler` 等会自动添加进来。

---

### 5. `Core/Src/main.c` / `Core/Inc/main.h`

**用户应用主文件。**

`main()` 函数的标准初始化流程：

```c
int main(void) {
    HAL_Init();            // 初始化 HAL 库，配置 SysTick，调用 HAL_MspInit()
    SystemClock_Config();  // 配置系统时钟（本项目：HSE 8MHz × 9 = 72MHz）
    MX_GPIO_Init();        // 初始化 GPIO（CubeMX 生成）
    // MX_XXX_Init();      // 其他外设初始化...

    while (1) {
        // 你的业务逻辑
    }
}
```

**当前时钟配置：**
- 时钟源：HSE（外部晶振 8MHz）
- PLL 倍频：× 9
- 系统时钟（SYSCLK）：**72MHz**
- APB1 总线：36MHz（UART2/3、SPI2、I2C 等）
- APB2 总线：72MHz（UART1、SPI1、GPIO 等）

`main.h` 的作用是统一包含 HAL 头文件，并声明 `Error_Handler()`，让所有源文件只需要 `#include "main.h"` 就能获得完整的类型和函数声明。

---

### 6. `Core/Src/syscalls.c`

**Newlib C 标准库的系统调用桩（Syscall Stubs）。**

裸机环境没有操作系统，但 C 标准库（Newlib）的某些函数（如 `printf`、`malloc`、`fopen`）需要调用底层 OS 接口。`syscalls.c` 提供这些接口的最简实现：

| 函数 | 说明 |
|------|------|
| `_write` | 标准输出，默认转发给 `__io_putchar()`，可接入 UART 实现 `printf` |
| `_read` | 标准输入，默认转发给 `__io_getchar()` |
| `_exit` | 程序退出，卡死在 `while(1)` |
| `_kill` / `_getpid` | 进程相关，在裸机上无意义，返回错误 |
| `_open` / `_close` 等 | 文件系统相关，全部返回失败 |

> **如需使用 `printf` 输出到串口**，只需在项目中实现 `__io_putchar()`，将字符发送到 UART 即可，无需修改 `syscalls.c`。

---

### 7. `Core/Src/sysmem.c`

**堆内存管理，提供 `_sbrk()` 实现。**

`malloc()`、`calloc()` 等动态内存分配函数依赖 `_sbrk()` 来向系统申请更多堆空间。`sysmem.c` 实现了一个安全版本：

```
RAM 布局：
┌──────────┬────────┬──────────────────────┬──────────────┐
│  .data   │  .bss  │      heap (堆)        │  MSP stack   │
│          │        │  ← _sbrk() 在此扩张  │  (栈，向下)  │
└──────────┴────────┴──────────────────────┴──────────────┘
           _end ↑                          ↑ _estack
```

关键保护：当堆扩张触碰到为栈预留的空间时，`_sbrk()` 返回 `ENOMEM` 错误，防止堆栈碰撞导致系统崩溃。

> **通常不需要修改。** 如果遇到 `malloc` 失败，考虑在链接脚本中增大 `_Min_Heap_Size`。

## 文件修改频率总结

| 文件 | 是否需要修改 | 修改场景 |
|------|-------------|---------|
| `startup_stm32f103c8tx.s` | 几乎不改 | 需要重定位向量表时 |
| `system_stm32f1xx.c` | 几乎不改 | 修改 HSE 晶振频率定义时 |
| `stm32f1xx_hal_msp.c` | **经常改** | 每次添加新外设时，CubeMX 自动生成 |
| `stm32f1xx_it.c` | **经常改** | 每次使能新的中断时，CubeMX 自动生成 |
| `main.c` | **经常改** | 业务逻辑开发 |
| `main.h` | 偶尔改 | 添加全局宏定义或引脚别名 |
| `syscalls.c` | 基本不改 | 实现 `printf` 重定向时 |
| `sysmem.c` | 基本不改 | 调整堆栈大小时修改链接脚本即可 |
